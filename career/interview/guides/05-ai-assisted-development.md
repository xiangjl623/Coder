## 一、AI 辅助编程

### 场景

用 Cursor、GitHub Copilot、Claude 等加速 boilerplate、测试、重构与文档；**不能**替代对正确性与安全的责任。

### 推荐工作流

```
1. 自己拆任务、定接口与约束
2. 给 AI 清晰上下文（文件、错误栈、期望行为）
3. 生成代码 → 人工 Review → 跑测试 / 静态分析
4. 提交前确保理解每一行关键逻辑
```

### Prompt 要点

| 原则 | 反例 |
|------|------|
| 说明语言、框架、版本 | 「写个排序」 |
| 给出输入输出示例 | 无边界条件 |
| 要求解释复杂度或 trade-off | 只要代码不要说明 |
| 分步：先设计再实现 | 一次生成整个系统 |

### 适合交给 AI 的任务

- 单元测试骨架、正则、数据类 CRUD
- 根据报错栈定位可能原因（需人工确认）
- 注释、README 初稿、API 文档草稿
- 语言间语法对照（C → C++ 迁移）

### 不应盲目依赖 AI 的任务

- 安全敏感：鉴权、加密、SQL 拼接
- 并发与锁的正确性
- 未经验证的性能关键路径
- 抄袭未知许可证的代码块

### 代码审查清单（AI 生成后必做）

- [ ] 编译 / 运行通过
- [ ] 边界与异常处理
- [ ] 无硬编码密钥
- [ ] 复杂度是否可接受
- [ ] 是否符合项目风格（见 [CONTRIBUTING.md](../../../CONTRIBUTING.md)）

## 二、LLM 应用基础

### 核心概念

| 概念 | 说明 |
|------|------|
| Prompt | 输入指令与上下文 |
| Token | 模型处理的最小文本单元；影响成本与长度上限 |
| 幻觉 | 模型自信地输出错误事实 |
| 上下文窗口 | 一次可处理的 token 上限 |

### RAG（检索增强生成）

```
用户问题 → 嵌入向量 → 检索知识库片段 → 拼入 Prompt → LLM 生成答案
```

**作用**：降低幻觉、接入私有/最新文档。  
**注意**：分块策略、召回率、引用溯源、权限隔离。

### Agent

LLM + **工具调用**（查 DB、调 API、跑代码）+ 规划循环。

```
用户目标 → Agent 规划步骤 → 调用工具 → 观察结果 → 继续或结束
```

**风险**：工具权限过大、无限循环、成本失控 → 需沙箱、步数上限、人工确认。

### 评测

- 准确率、延迟、成本
- 人工抽检 + 黄金集回归
- 对抗性用例（诱导幻觉、越权）

## 三、面试策略

### 工程岗：如何谈 AI 使用

**推荐表述**：

> 我用 Cursor 加速写测试和样板代码，但核心算法和业务规则自己设计；合并前会跑测试并 Review diff，确保理解变更。

**避免**：

> 我所有代码都是 AI 写的，我不用看。

### 可能的新考点

| 方向 | 示例问题 |
|------|----------|
| AI 协作 | 你如何验证 AI 生成的代码？ |
| LLM 基础 | RAG 和微调分别解决什么问题？ |
| 产品思维 | 如何降低幻觉对用户的影响？ |
| 安全 | Prompt 注入是什么？如何防护？ |

### AI 应用岗 vs 传统后端

| | 传统后端 | AI 应用 / LLM 工程 |
|---|----------|---------------------|
| 重点 | API、DB、分布式 | Prompt、RAG、评测、推理成本 |
| 数据 | 结构化事务 | 向量库、文档 pipeline |
| 风险 | 一致性、安全 | 幻觉、隐私、模型依赖 |

两者都需扎实工程能力；AI 岗额外考察 **模型边界与评测意识**。

### 与经典面试技巧的结合

沿用 [01 面试准备](01-interview-preparation.md) 的结构：

1. **是什么** — 例如 RAG 的定义  
2. **作用** — 为何不用纯 Prompt  
3. **注意点** — 分块、权限、评测  
4. **结合项目** — 若做过 AI 功能，讲清你的决策与验证

## 四、小型案例：用 AI 辅助完成链表练习

**场景**：复习 [fundamentals/04-data-structures/](../../../fundamentals/04-data-structures/) 链表章。

1. **自己**：画出反转链表指针变化图  
2. **AI**：根据你的图生成 C 实现草稿  
3. **你**：手工 trace `[1,2,3]`，发现 off-by-one，修正  
4. **AI**：生成 3 个边界测试用例  
5. **你**：运行测试，写入 [exercises/](../../../fundamentals/04-data-structures/exercises/)

这样既提速，又保留学习深度 — 面试时可完整讲述过程。

## 本节小结

- AI 用于提效；验证、安全、架构必须人负责
- RAG = 检索 + 生成；Agent = 规划 + 工具
- 面试诚实描述协作方式，展示 Review 与评测思维
