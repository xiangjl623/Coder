# 面试准备

> **难度**：入门 | **阅读时间**：约 25 分钟 | **前置**：无 | **标签**：`interview` `resume` `behavioral`

## 学习目标

- [ ] 能用 STAR 法则写清 1～2 个项目经历
- [ ] 规划笔试 → 技术面 → HR 面的时间线与复盘方式
- [ ] 掌握技术题与行为题的通用答题结构

---

## 一、简历与作品集

### 场景

面试官在 30 秒内决定是否继续看你的项目。简历不是技术名词清单，而是**可验证能力的索引**。

### 原则

| 原则 | 说明 |
|------|------|
| 一条经历一个亮点 | 每段项目写清：背景、你的职责、技术选型、**量化结果** |
| 可点击验证 | 附 GitHub 仓库、在线 Demo、README 或设计文档链接 |
| 匹配 JD | 按岗位调整项目顺序，最相关的放前面 |
| 诚实 | 写「参与」就不写「主导」；AI 辅助的部分应能在面试中说清 |

### STAR 模板

```
Situation  — 什么业务/课程背景？
Task       — 你要解决什么问题？
Action     — 你具体做了什么（技术 + 协作）？
Result     — 结果如何量化？（性能、用户数、bug 率、交付周期）
```

**反例**：「熟悉 Java、Spring、MySQL、Redis、Kafka、Docker、K8s……」—— 无项目支撑时等于无效信息。

**正例**：

> 学生管理系统（C++，个人项目）  
> 用 vector + 文件持久化实现增删改查；通过封装 `StudentManager` 分离 UI 与业务；单元测试覆盖核心 CRUD，README 含编译与运行说明。  
> 仓库：[languages/cpp/examples/chapter14/](../../../languages/cpp/examples/chapter14/)

### 项目筛选

优先保留能讲清 **为什么这样设计、遇到过什么坑、如何验证** 的项目。课程大作业若只完成 CRUD，可强调异常处理、代码结构或测试，而非堆框架名。

## 二、面试流程与时间规划

### 典型流程

```
投递 → 简历筛选 → 笔试/OA → 1～3 轮技术面 → HR/综合面 → Offer
```

| 轮次 | 常见形式 | 准备重点 |
|------|----------|----------|
| 笔试 | 选择题 + 编程题 | [03 数据结构与算法](03-data-structures-and-algorithms.md)、[02 计算机基础](02-cs-fundamentals.md) |
| 一面 | 基础 + 项目 | 八股 + 项目深挖 |
| 二面 | 项目 + 系统设计 | [04 系统设计与工程实践](04-system-design-and-engineering.md) |
| HR 面 | 行为 + 薪资 | 本章第三节 |

### 时间线建议

经典面试合集强调：**早准备、每天几题、按月积累**，而非临阵抱佛脚。

| 阶段 | 建议 |
|------|------|
| 提前 2～3 个月 | 确定目标岗位；复习 [fundamentals/](../../../fundamentals/) 数据结构与算法 |
| 提前 1 个月 | 刷高频题；整理 2 个项目的故事版（各 3 分钟 / 10 分钟两版） |
| 提前 1 周 | 查公司技术栈；模拟自我介绍；准备 3 个问面试官的问题 |
| 每场结束后 24h 内 | 复盘：考了什么、哪里卡壳、待补知识点 |

### 复盘模板

```markdown
## 公司 / 岗位 / 日期
- 题型：八股 / 算法 / 设计 / 行为
- 我答得好的：
- 卡壳的题：（查资料后补答案链接）
- 下次改进：
```

## 三、行为面试

### 高频问题

- 介绍一个你最有挑战的项目
- 与同事/同学意见不合怎么办
- 说一次失败经历及收获
- 为什么选这个岗位 / 公司
- 3～5 年职业规划

### STAR 回答示例（协作冲突）

**场景**：课程小组中两人对模块划分意见不一。  
**任务**：在 deadline 前完成可运行的交付物。  
**行动**：列出接口清单对齐边界；用 Git 分支各自开发；每日 15 分钟 sync。  
**结果**：按期提交，README 中记录了分工与接口约定。

### 反例

- 只抱怨队友不配合，不提自己的协调动作
- 把课程作业说成「千万级用户线上系统」
- 对薪资一问三不知或漫天要价（应届生宜了解行情、留合理区间）

## 四、答题通用结构（技术题同样适用）

来自经典面试总结的答题法，至今仍然有效：

```
1. 是什么     — 先给定义或结论
2. 作用/原理  — 为什么需要、怎么工作
3. 注意点     — 边界、坑、与你项目的联系（展现深度）
4. 思路过程   — 现场题要讲推导，不要只报答案
```

**原则**：

- 层次分明、条理清晰
- 题目模糊时，**先澄清再作答**（例如「您问的是进程还是线程层面的并发？」）
- 把面试官当作未来同事，交流而非背稿

## 本节小结

- 简历重可验证产出，STAR + 量化结果
- 按轮次分配复习精力，每场必复盘
- 行为面用 STAR；技术面用「是什么 → 原理 → 注意点 → 思路」

3. 准备 1 分钟与 3 分钟两版自我介绍
