## 一、系统设计方法论

### 四步框架

```
1. 需求澄清   — 功能 / 非功能（QPS、延迟、一致性、可用性）
2. 容量估算   — DAU、读写比、存储增长（Back-of-envelope）
3. 高层架构   — 客户端、网关、服务、缓存、DB、MQ
4. 深入与权衡 — 瓶颈、单点、扩展、一致性方案
```

### 非功能指标（先问清）

| 指标 | 示例问题 |
|------|----------|
| 延迟 | P99 多少 ms 可接受？ |
| 可用性 | 能否接受短暂只读？ |
| 一致性 | 强一致还是最终一致？ |
| 规模 | 日活、峰值 QPS、数据保留多久？ |

### 常用扩展手段

- **垂直扩展**：更强机器（有上限）
- **读写分离**：主写从读
- **分库分表**：按 user_id hash
- **缓存**：Redis 挡热点
- **异步**：MQ 削峰填谷
- **CDN**：静态资源边缘分发

## 二、典型场景

### 短链服务

**需求**：长 URL → 短码；重定向；可选统计点击。

**要点**：

- 短码：Base62 自增 ID 或 hash + 冲突处理
- 读多写少 → 缓存短码 → 长 URL
- 302 重定向；计数可异步写 MQ

### Feed 时间线

**推模式**：写时 fan-out 到粉丝收件箱（大 V 粉丝多时用拉/混合）。  
**拉模式**：读时聚合关注列表（读压力大）。

**权衡**：Twitter 类混合 — 普通用户推，明星拉。

### 秒杀

**难点**：瞬时超高并发、超卖、DB 压力。

**思路**：

```
请求 → 网关限流 →  Redis 预减库存 → 异步 MQ 下单 → DB 落库
```

- 库存预热到 Redis；Lua 保证原子减
- 失败快速返回，避免打穿 DB
- 仍需最终对账与防刷

### 文件存储

元数据（DB）+ 对象存储（S3/OSS）；大文件分片上传；CDN 加速下载。

## 三、现代后端

### REST 设计

| 方法 | 语义 | 幂等 |
|------|------|------|
| GET | 查询 | 是 |
| POST | 创建 | 否 |
| PUT | 全量更新 | 是 |
| PATCH | 部分更新 | 否 |
| DELETE | 删除 | 是 |

资源 URI 用名词：`/users/{id}/orders`。

### 认证

- **Session + Cookie**：服务端存会话，传统 Web
- **JWT**：无状态，适合 API；注意过期、刷新、密钥轮换
- **OAuth2**：第三方登录授权框架

### 幂等与限流

- 幂等键：`Idempotency-Key` 头 + 服务端去重表
- 限流：令牌桶 / 漏桶；网关或 sidecar 统一实施

### gRPC vs REST

| | REST/JSON | gRPC |
|---|-----------|------|
| 协议 | HTTP | HTTP/2 + Protobuf |
| 场景 | 公开 API、浏览器 | 服务间高性能 RPC |

## 四、分布式与中间件

### CAP（口述）

分区发生时，**C**（一致）与 **A**（可用）往往权衡；**P**（分区容忍）在分布式系统中通常必须接受。

### 消息队列

| 能力 | 用途 |
|------|------|
| 解耦 | 订单创建 → 通知、积分、物流各自消费 |
| 削峰 | 秒杀订单异步写库 |
| 最终一致 | 跨服务数据同步 |

需说明：至少一次 / 恰好一次 / 重复消费与幂等。

### 缓存问题

| 问题 | 思路 |
|------|------|
| 穿透 | 布隆过滤器 / 空值缓存 |
| 击穿 | 热点 key 互斥重建 |
| 雪崩 | 过期时间加随机、多级缓存 |

### 熔断降级

下游超时或错误率超阈值 → 快速失败或返回兜底，防止级联故障（配合 Hystrix / Sentinel 等概念）。

## 五、云原生与工程化

### 架构演进（对照经典 C/S vs B/S）

| | C/S | B/S |
|---|-----|-----|
| 客户端 | 专用程序 | 浏览器 |
| 部署 | 每台安装升级 | 服务端升级即可 |
| 现代对应 | 桌面 / 移动 App | Web + API；微服务 + 前端 SPA |

当前主流：**B/S + 移动端 + 后端微服务 + 容器编排**。

### 容器与 K8s（概念级）

```
Docker：镜像打包依赖 → 容器运行实例
K8s：Deployment 管理副本；Service 暴露访问；Pod 最小调度单元
```

### CI/CD

```
git push → CI 跑测试/构建 → 镜像推送 → CD 滚动发布
```

关联 [engineering/01-git-and-collaboration/](../../../engineering/01-git-and-collaboration/)。

### 可观测性

- **日志**：集中收集（ELK 等）
- **指标**：QPS、延迟、错误率（Prometheus）
- **链路**：分布式 trace（Jaeger 等）

## 六、答题示例：设计「短链 + 1000 QPS 读」

1. **需求**：读远大于写；短码 7 字符；99.9% 可用。
2. **估算**：1000 QPS 读 → 单 Redis 集群可挡；写 QPS 低，MySQL 足够。
3. **架构**：LB → API → Redis（读）→ MySQL（写 + 持久化）。
4. **深入**：短码生成、缓存 miss 回源、DB 主从、监控告警。
