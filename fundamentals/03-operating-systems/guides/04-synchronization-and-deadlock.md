# 同步与死锁

> **难度**：进阶 | **阅读时间**：约 20 分钟 | **前置**：[进程与线程](01-process-and-thread.md) | **标签**：`mutex` `deadlock`

## 学习目标

- [ ] 说明竞态条件与互斥的必要性
- [ ] 列举死锁四个必要条件
- [ ] 了解信号量、互斥锁的基本用法

## 竞态条件

多线程同时读写共享变量，结果依赖调度顺序：

```c
/* 两个线程各执行 count++ 一万次，结果可能 < 20000 */
count++;
/* 读-改-写 非原子 */
```

**解决**：互斥 — 同一时刻只有一个线程进入临界区。

## 互斥手段

| 机制 | 说明 |
|------|------|
| 互斥锁 mutex | lock / unlock |
| 信号量 semaphore | 计数资源，P/V 操作 |
| 条件变量 | 等待某条件成立（配合 mutex） |

**原则**：锁粒度宜小、持有时间宜短；固定加锁顺序防死锁。

## 死锁

四个**必要条件**（同时满足才可能死锁）：

1. **互斥**：资源独占
2. **占有且等待**：持有一个等另一个
3. **不可抢占**：只能主动释放
4. **循环等待**：形成环

**经典例子**：线程 A 锁 1 等锁 2；线程 B 锁 2 等锁 1。

### 预防与处理

- **预防**：固定加锁顺序、一次性申请所有锁、超时放弃
- **检测与恢复**：银行家算法（了解）
- **鸵鸟策略**：极少发生则不管（不推荐生产）

## 同步 vs 异步

| | 同步 | 异步 |
|---|------|------|
| 调用方 | 等待完成 | 提交后继续 |
| 例子 | 阻塞 read | 回调、Future、epoll |

I/O 密集常用异步提高吞吐；CPU 密集用多线程/多进程。

## 面试关联

- `sleep()` vs `wait()`（Java 语境）：sleep 不释放锁；wait 释放锁并等待 notify
- `synchronized` vs `Lock`：自动 vs 显式、可中断/公平

## 本节小结

- 共享数据必须同步
- 死锁四条件；固定加锁顺序是实用预防
- 锁范围越小越好
